1. PENDAHULUAN


1.1 Tujuan Dokumen


Dokumen ini mendeskripsikan arsitektur perangkat lunak MikoMart POS secara menyeluruh, meliputi: decomposisi komponen, pola arsitektur, keputusan desain (ADR), strategi deployment, dan arsitektur keamanan. Dokumen ini ditujukan untuk engineer, QA, dan stakeholder teknis.


1.2 Ruang Lingkup Arsitektur


Arsitektur mencakup:

  • Frontend: React 18 SPA (Single Page Application) dengan kemampuan offline-first
  • Backend API: PHP Laravel 11 (REST API)
  • Real-time Service: Node.js 18 + WebSocket (notifikasi live)
  • Database Lokal: SQLite 3.x dengan mode WAL + enkripsi SQLCipher
  • Database Server: SQLite/MySQL 8.x (server pusat)
  • Infrastruktur: On-premise server di toko + internet untuk integrasi payment

3. KOMPONEN ARSITEKTUR


3.1 Frontend (Client Layer)


Teknologi: React 18, TypeScript, Vite, Zustand (state management), Workbox (Service Worker)


KomponenDeskripsi
React SPASingle-Page Application, dioptimalkan untuk tablet touchscreen. Semua halaman dirender client-side.
Service Worker (Workbox)Mengabstraksi koneksi — mengantri request saat offline dan mengirimnya saat online. Cache strategy: Network-First untuk API, Cache-First untuk static assets.
SQLite Lokal (SQLCipher)Database lokal yang dienkripsi. Menyimpan transaksi, produk, dan konfigurasi untuk operasi offline. Diakses via sql.js atau absurd-sql.
Zustand StoreState management global: cart, user session, sync status, notification queue. Persisted ke localStorage (non-sensitive) dan SQLite (sensitive).

3.2 API Gateway Layer


Teknologi: Nginx 1.24+, SSL/TLS (Let's Encrypt / self-signed untuk on-premise)


KomponenDeskripsi
Nginx Reverse ProxyMeneruskan semua request ke Laravel atau Node.js berdasarkan path (/api/ → Laravel, /ws/ → Node.js). Terminasi TLS dilakukan di sini.
Rate LimiterNginx limit_req_zone — max 100 req/menit per IP untuk endpoint publik; 300 req/menit untuk klien terautentikasi.
Security HeadersContent-Security-Policy, Strict-Transport-Security, X-Frame-Options: DENY, X-Content-Type-Options: nosniff.
Webhook EndpointEndpoint khusus /api/v1/webhooks/payment — hanya menerima POST dari IP whitelist payment gateway.

3.3 Services Layer


Teknologi: PHP 8.2 (Laravel 11), Node.js 18 (ws library), Laravel Queues (Redis)


KomponenDeskripsi
Laravel APIController-Service-Repository pattern. Semua query menggunakan Eloquent ORM. Validasi input via FormRequest.
Node.js WebSocketServer-Sent Events / WebSocket untuk notifikasi real-time: status pembayaran QRIS, void approval notification, stok minimum alert.
Background JobsSyncJob memproses antrian sinkronisasi. AuditLogPurger menghapus log > 30 hari (via cron). StockMinimumChecker berjalan setiap 15 menit.
Domain ServicesBusiness logic dikemas dalam Service classes yang dapat diuji secara unit.

3.4 Data Layer


KomponenTeknologiDeskripsi
SQLite LokalSQLite 3.x + SQLCipherDatabase terenkripsi per perangkat kasir. Mode WAL untuk keandalan.
MySQL ServerMySQL 8.0+Database pusat. Semua data yang telah tersinkronisasi tersimpan di sini.
RedisRedis 7+Backend antrian Laravel Queue dan cache session.
File StorageLocal diskPDF struk dan file ekspor disimpan di luar web root (/storage/app/private/).

6. SECURITY ARCHITECTURE


6.1 Threat Model (STRIDE)


STRIDE adalah framework ancaman Microsoft (Spoofing, Tampering, Repudiation, Information Disclosure, Denial of Service, Elevation of Privilege).


6.1.1 SPOOFING (Pemalsuan Identitas)


IDSkenario AncamanKomponen TargetMitigasi
T-SP-01Penyerang menebak password kasir yang lemahAuth endpointArgon2id hashing; account lockout 5x gagal; password minimum 8 karakter
T-SP-02Token JWT dicuri dan digunakan kembaliClient ↔ APIToken expiry 60 menit; HTTPS wajib; HttpOnly cookie; refresh token invalidation
T-SP-03Webhook palsu dikirim untuk konfirmasi pembayaran fiktifWebhook endpointHMAC-SHA512 signature verification wajib; IP whitelist payment gateway

6.1.2 TAMPERING (Manipulasi Data)


IDSkenario AncamanKomponen TargetMitigasi
T-TM-01Kasir memanipulasi payload transaksi di client-sideAPI endpointValidasi semua input di server-side (FormRequest Laravel); tidak percaya data dari client
T-TM-02Manipulasi diskon melebihi 30% via API langsungTransactionControllerValidasi server-side: discount_percent ≤ 30; price_final ≥ price_buy; audit log setiap override
T-TM-03Modifikasi data offline di SQLite sebelum syncSyncControllerVerifikasi checksum payload sync; verifikasi user_id konsisten dengan token; conflict detection
T-TM-04SQL Injection melalui input pencarian produkProductControllerEloquent ORM dengan prepared statements wajib; input sanitization

6.1.3 REPUDIATION (Penyangkalan)


IDSkenario AncamanKomponen TargetMitigasi
T-RP-01Kasir menyangkal telah memberikan diskon berlebihAudit LogSetiap diskon/override wajib tercatat di audit_logs dengan user_id, timestamp, nilai sebelum-sesudah
T-RP-02Admin menyangkal telah mengubah harga produkAudit LogSemua CRUD produk dan perubahan harga dicatat di audit log (value_before, value_after)
T-RP-03Sengketa pembayaran — pelanggan klaim tidak bayarpayment_recordsSimpan gateway_response JSON lengkap; webhook timestamp; gateway_transaction_id

6.1.4 INFORMATION DISCLOSURE (Kebocoran Informasi)


IDSkenario AncamanKomponen TargetMitigasi

6.1.5 DENIAL OF SERVICE (Penolakan Layanan)


IDSkenario AncamanKomponen TargetMitigasi
T-DS-01Flood request ke API dari perangkat kasirAPI GatewayRate limiter Nginx: 300 req/menit per IP terotentikasi; 100 req/menit untuk unauthenticated
T-DS-02Antrian sinkronisasi penuh menyebabkan DB overloadSyncJobBatching sync: max 50 records per job; back-pressure mechanism; retry limit 3x
T-DS-03Upload laporan sangat besar memblokir serverReportControllerPagination wajib; ekspor besar diproses as background job dan diunduh setelah selesai

6.1.6 ELEVATION OF PRIVILEGE (Eskalasi Hak Akses)


IDSkenario AncamanKomponen TargetMitigasi

7. ARCHITECTURE DECISION RECORDS (ADR)


ADR-001: Offline-First dengan SQLite + Service Worker


AspekKeputusan
Konteks4 kasir aktif di toko; koneksi internet tidak dijamin stabil
KeputusanImplementasi offline-first: React SPA + Service Worker + SQLite lokal
AlternatifFull online (langsung ke server)
AlasanTransaksi tunai harus tetap berjalan saat offline; downtime server tidak boleh mempengaruhi operasi kasir
KonsekuensiKompleksitas sinkronisasi dan conflict resolution meningkat; SQLCipher diperlukan untuk keamanan data lokal

ADR-002: Laravel (PHP) untuk Backend API


AspekKeputusan
KonteksPilihan stack backend — PHP atau Node.js penuh
KeputusanLaravel 11 (PHP 8.2) untuk REST API utama
AlternatifExpress.js atau NestJS (Node.js penuh)
AlasanLaravel memiliki ekosistem lengkap: Eloquent ORM, FormRequest validation, Queue, Middleware, Sanctum auth; developer lebih banyak berbasis PHP di Indonesia
KonsekuensiNode.js hanya digunakan untuk WebSocket real-time (lebih efisien untuk event streaming)

ADR-003: Node.js WebSocket untuk Real-time Notification


AspekKeputusan
KonteksStatus pembayaran QRIS dan void approval perlu notifikasi real-time
KeputusanNode.js WebSocket server terpisah dari Laravel API
AlternatifLaravel Broadcasting dengan Pusher atau SSE
AlasanWebSocket lebih efisien untuk banyak koneksi simultan; Node.js event-loop lebih optimal untuk I/O intensif
KonsekuensiDiperlukan satu service Node.js tambahan; Nginx harus route /ws/* ke Node.js

ADR-004: Two-Step Void/Return Approval


AspekKeputusan
KonteksRisiko penyalahgunaan void oleh kasir
KeputusanKasir hanya mengajukan; Supervisor/Admin yang menyetujui
AlternatifKasir mandiri (sesuai jawaban awal)
AlasanMencegah fraud; memberikan oversight kepada manajemen; setiap langkah tercatat di audit log
KonsekuensiKasir perlu menunggu approval — diperlukan SLA void approval internal (target: ≤ 5 menit)

ADR-005: Gitflow sebagai Branching Strategy


AspekKeputusan
KonteksTim kecil (≤5 developer); perlu track milestone dan hotfix
KeputusanGitflow dengan branch: main, develop, feature/, release/, hotfix/*
AlternatifTrunk-based development
AlasanGitflow lebih sesuai untuk proyek dengan milestone tetap dan siklus rilis terencana (seperti jadwal UAT); lebih mudah di-enforce untuk tim baru
KonsekuensiLebih banyak branch management overhead vs trunk-based; butuh disiplin untuk tidak delay merge
